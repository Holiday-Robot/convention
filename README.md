# Convention

코드 또는 프레임워크 관련된 코드 스니펫, 예제, 모범 사례를 모아놓은 저장소입니다.
이 저장소의 목표는 Holiday Robotics가 작성한 코드에 일관된 스타일과 구조를 유지하여 가독성과 유지보수성을 향상시키는 것입니다.
또한, Holiday Robotics의 개발자들이 코드 작성 시 참고할 수 있는 가이드라인을 제공하여 코드 품질을 높이는 것을 목표로 합니다.

> [!WARNING]
> 현재 모든 문서의 내용은 초안이며 강제적이지 않습니다.

## 기본 원칙

이 저장소의 모든 가이드라인은 아래 원칙들을 기반으로 작성되었습니다.
개별 규칙이 원칙과 충돌하는 것처럼 보인다면, 원칙의 의도를 우선합니다.

### 1. 단순성 (Simplicity)

규칙은 기억하고 적용하기 쉬워야 합니다.
예외가 많거나 조건이 복잡한 규칙은 그 자체로 혼란을 유발합니다.
규칙을 단순하게 유지할 수 없다면, 규칙을 두지 않는 것이 낫습니다.

### 2. 커뮤니티 표준 우선 (Community Standards First)

언어나 프레임워크 커뮤니티에 공식적으로 정립된 스타일 가이드가 있다면, 이를 가이드라인의 기초로 삼습니다.
커뮤니티 표준을 벗어나는 규칙은 명확한 이유가 있을 때에만 정의하며, 그 이유를 문서에 명시합니다.

### 3. 자동화 우선 (Automation First)

스타일에 관한 결정은 사람이 아닌 도구가 내립니다.
Formatter와 Linter가 강제할 수 있는 규칙만을 스타일 규칙으로 정의하며, 도구로 검증할 수 없는 규칙은 스타일이 아닌 설계 지침으로 분류합니다.

#### 3.1. 도구 선택

가이드라인은 스타일 정의의 기초가 된 도구를 문서에 명시적으로 언급해야하며, 도구는 언어의 커뮤니티에서 널리 사용되고 지원되는 것을 선택해야 합니다.

### 4. 최소 놀람 (Least Surprise)

코드는 독자의 기대를 벗어나지 않아야 합니다.
언어나 프레임워크의 관용적인 방식(idiom)을 따르고, 독창적인 표현보다 예측 가능한 표현을 선택합니다.
영리한 코드보다 지루하지만 명확한 코드가 낫습니다.


## 기여

### 언어 사용자

"언어 사용자"의 정의는 추후 팀 내 합의를 통해 결정합니다.

### Merge 기준

가이드라인의 변경은 그 영향이 지속적으로 남는 만큼, 충분한 검토를 거쳐야 합니다.

#### 프리즈 이전

초기 가이드라인을 수립하는 단계에서는, 충분한 토론을 거친 변경에 한해 해당 언어 사용자 최소 2명의 승인을 받으면 merge할 수 있습니다.

#### 프리즈 이후

가이드라인이 합의된 프리즈 상태에 들어간 이후에는, Holiday Robotics 엔지니어 중 해당 언어 사용자의 **66% 이상**의 동의를 받아야만 merge할 수 있습니다.
